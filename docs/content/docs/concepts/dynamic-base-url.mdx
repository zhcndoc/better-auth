---
title: 动态基础 URL
description: 配置 Better Auth 以支持多个域名和预览部署。
---

在部署到如 Vercel 等平台时，您的应用可能会同时运行在多个 URL 上：
- 自定义域名（例如，`myapp.com`，`www.myapp.com`）
- 预览部署（例如，`my-app-abc123.vercel.app`）
- 分支预览（例如，`feature-branch.myapp.com`）

Better Auth 支持使用**允许列表（allowlist）方法**动态解析基础 URL，从每个传入请求中确定正确的 URL。

## 基本用法

将 `baseURL` 配置为一个对象，包含 `allowedHosts` 数组：

```ts title="auth.ts"
import { betterAuth } from "better-auth"

export const auth = betterAuth({
    baseURL: {
        allowedHosts: [
            "myapp.com",
            "www.myapp.com",
            "*.vercel.app",
        ],
    },
})
```

当请求到来时，Better Auth 会从 `x-forwarded-host` 或 `host` 头中提取主机，并将其与允许列表校验。如果匹配，该 URL 将用作请求的基础 URL。

## 通配符模式

`allowedHosts` 数组支持通配符模式：

| 模式 | 匹配 |
|---------|---------|
| `myapp.com` | 仅完全匹配 |
| `*.vercel.app` | vercel.app 的任意子域 |
| `preview-*.myapp.com` | 以 `preview-` 开头的子域 |
| `localhost:*` | 任何端口的 localhost |

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: [
            "myapp.com",           // 生产环境
            "*.vercel.app",        // 所有 Vercel 预览
            "preview-*.myapp.com", // 自定义预览子域
            "localhost:3000",      // 本地开发
        ],
    },
})
```

## 备用 URL

默认情况下，如果传入的主机不匹配任何 `allowedHosts` 中的模式，Better Auth 会抛出错误。你可以提供备用 URL：

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: ["myapp.com", "*.vercel.app"],
        fallback: "https://myapp.com",
    },
})
```

配置备用 URL 后：
- 匹配的主机使用派生的 URL
- 不匹配的主机使用备用 URL（不会抛出错误）

<Callout type="warn">
请谨慎使用备用 URL。 Silent fallback 可能掩盖配置错误。请考虑是否对您的用例更适合采用抛出错误（默认）。
</Callout>

## 协议配置

控制构建 URL 时使用的协议：

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: ["myapp.com", "localhost:3000"],
        protocol: "https", // 始终使用 HTTPS
    },
})
```

| 值 | 行为 |
|-------|----------|
| `"https"` | 始终使用 HTTPS |
| `"http"` | 始终使用 HTTP |
| `"auto"` | 从 `x-forwarded-proto` 头部推断，然后是请求 URL；若两者均不可用，则默认为 HTTPS |
| 不设置 | 同 `"auto"` |

<Callout type="info">
`protocol` 设置控制每个请求**构建 URL 的方式**。对于 Cookie 的 `Secure` 标志，Better Auth 使用 `protocol: "https"` → 安全，`protocol: "http"` → 不安全，对于 `"auto"` 或未设置 → 退回到 `NODE_ENV === "production"`。你也可以通过 `advanced.useSecureCookies` 进行覆盖。
</Callout>

对于混合环境（本地开发 + 生产），你可以使用环境变量：

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: ["myapp.com", "localhost:3000"],
        protocol: process.env.NODE_ENV === "development" ? "http" : "https",
    },
})
```

## 可信来源集成

配置 `allowedHosts` 时，Better Auth 会自动将它们添加到 `trustedOrigins` 中，意味着：

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: ["myapp.com", "*.vercel.app"],
    },
})

// 自动添加到 trustedOrigins：
// - "https://myapp.com"
// - "https://*.vercel.app"
```

你无需在两个配置中重复允许的主机。

## 静态基础 URL（向后兼容）

传统的静态字符串配置仍然有效：

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: "https://myapp.com",
})
```

当你的应用仅运行在单一确定的 URL 上时，推荐使用此方式。

## 常见模式

### Vercel 部署

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: [
            "myapp.com",        // 生产自定义域
            "www.myapp.com",    // WWW 变体
            "*.vercel.app",     // 所有预览部署
        ],
    },
})
```

### 开发 + 生产

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: [
            "localhost:3000",
            "localhost:5173",
            "myapp.com",
            "*.vercel.app",
        ],
        protocol: process.env.NODE_ENV === "development" ? "http" : "https",
    },
})
```

### 多生产域名

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: [
            "myapp.com",
            "myapp.co.uk",
            "myapp.eu",
        ],
    },
})
```

## 跨子域 Cookie

如果需要在子域间共享 Cookie，可以开启 `crossSubDomainCookies`，且不指定静态 `domain`。Better Auth 会自动从每个请求解析的主机推断 Cookie 域。

```ts title="auth.ts"
import { betterAuth } from "better-auth"

export const auth = betterAuth({
    baseURL: {
        allowedHosts: [
            "auth.example1.com",
            "auth.example2.com",
        ],
        protocol: "https",
    },
    advanced: {
        crossSubDomainCookies: {
            enabled: true,
        },
    },
})
```

配置后：
- 对 `auth.example1.com` 的请求，其 Cookie 带 `Domain=auth.example1.com`
- 对 `auth.example2.com` 的请求，其 Cookie 带 `Domain=auth.example2.com`

如需对所有主机使用相同的域名，可以显式设置 `domain`：

```ts title="auth.ts"
import { betterAuth } from "better-auth"

export const auth = betterAuth({
    baseURL: {
        allowedHosts: ["auth.example1.com", "auth.example2.com"],
        protocol: "https",
    },
    advanced: {
        crossSubDomainCookies: {
            enabled: true,
            domain: ".example.com", // 不论主机均使用此域
        },
    },
})
```

<Callout type="info">
使用静态 `baseURL` 字符串时，`crossSubDomainCookies` 在初始化时从该字符串派生域名。使用动态基础 URL 时，域名根据每个请求解析的主机动态计算。更多跨子域 Cookie 详情见 [Cookies](/docs/concepts/cookies#cross-subdomain-cookies) 文档。
</Callout>

## 安全注意事项

<Callout type="info">
动态基础 URL 采用**允许列表（allowlist）策略**确保安全。仅接受 `allowedHosts` 中明确定义的主机。
</Callout>

1. **允许列表为必填** — 不接受任意主机
2. **头部验证** — 对 `x-forwarded-host` 和 `host` 头进行净化处理后使用
3. **显式模式** — 无自动平台检测；须手动添加每个模式
4. **明确错误** — 未知主机会抛出描述性错误（除非配置了 fallback）

这不同于一些自动信任平台或接受任意头部值的其他鉴权库。Better Auth 优先考虑安全。  

## 配置参考

| 属性 | 类型 | 默认 | 描述 |
|----------|------|---------|-------------|
| `allowedHosts` | `string[]` | 必填 | 允许的主机模式列表 |
| `fallback` | `string` | - | 主机不匹配时使用的 URL |
| `protocol` | `"http" \| "https" \| "auto"` | `"auto"` | URL 构建使用的协议。`"auto"` 从请求头派生，默认 HTTPS |
