---
title: 动态基础 URL
description: 配置 Better Auth 以支持多个域名和预览部署。
---

当部署到 Vercel 这类平台时，你的应用可能同时运行在多个 URL 上：

* 自定义域名（例如：`myapp.com`、`www.myapp.com`）
* 预览部署（例如：`my-app-abc123.vercel.app`）
* 分支预览（例如：`feature-branch.myapp.com`）

Better Auth 支持使用 **允许列表（allowlist）方法** 动态解析基础 URL，从每个传入请求中确定正确的 URL。

## 基本用法

将 `baseURL` 配置为一个对象，并包含 `allowedHosts` 数组：

```ts title="auth.ts"
import { betterAuth } from "better-auth"

export const auth = betterAuth({
    baseURL: {
        allowedHosts: [
            "myapp.com",
            "www.myapp.com",
            "*.vercel.app",
        ],
    },
})
```

当请求到来时，Better Auth 会从 `x-forwarded-host` 或 `host` 头中提取主机，并将其与允许列表进行校验。如果匹配，该 URL 将用作请求的基础 URL。

## 通配符模式

`allowedHosts` 数组支持通配符模式：

| Pattern               | Matches                             |
| --------------------- | ----------------------------------- |
| `myapp.com`           | 仅精确匹配                          |
| `*.vercel.app`        | vercel.app 的任意子域名             |
| `preview-*.myapp.com` | 以 `preview-` 开头的子域名          |
| `localhost:*`         | 任意端口的 localhost                |

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: [
            "myapp.com",           // 生产环境
            "*.vercel.app",        // 所有 Vercel 预览
            "preview-*.myapp.com", // 自定义预览子域
            "localhost:3000",      // 本地开发
        ],
    },
})
```

## 备用 URL

默认情况下，如果传入的主机未匹配到 `allowedHosts` 中的任何模式，Better Auth 会抛出错误。你可以提供备用 URL：

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: ["myapp.com", "*.vercel.app"],
        fallback: "https://myapp.com",
    },
})
```

配置了备用 URL 后：

* 匹配的主机使用派生的 URL
* 不匹配的主机使用备用 URL（不会抛出错误）

<Callout type="warn">
  小心使用备用 URL。静默回退可能会掩盖配置错误。请根据你的使用场景判断抛出错误（默认行为）是否更合适。
</Callout>

## 协议配置

控制构建 URL 时使用的协议：

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: ["myapp.com", "localhost:3000"],
        protocol: "https", // 始终使用 HTTPS
    },
})
```

| 值       | 行为                                                                                            |
| -------- | --------------------------------------------------------------------------------------------------- |
| `"https"` | 始终使用 HTTPS                                                                                    |
| `"http"`  | 始终使用 HTTP                                                                                     |
| `"auto"`  | 从 `x-forwarded-proto` 头推导，再从请求 URL 推导；若都不可用，默认使用 HTTPS                         |
| 未设置    | 与 `"auto"` 相同                                                                                 |

<Callout type="info">
  `protocol` 设置控制 **URL 的构造方式**。对于 Cookie 的 `Secure` 标志，Better Auth 的处理如下：
  - `protocol: "https"` → 安全
  - `protocol: "http"` → 不安全
  - `"auto"` 或未设置 → 回退到 `NODE_ENV === "production"`
  你可以通过 `advanced.useSecureCookies` 始终覆盖此行为。
</Callout>

对于混合环境（本地开发 + 生产），可以使用环境变量：

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: ["myapp.com", "localhost:3000"],
        protocol: process.env.NODE_ENV === "development" ? "http" : "https",
    },
})
```

## 可信来源集成

配置 `allowedHosts` 时，Better Auth 会自动将它们添加到 `trustedOrigins` 中，这意味着：

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: ["myapp.com", "*.vercel.app"],
    },
})

// 自动添加到 trustedOrigins：
// - "https://myapp.com"
// - "https://*.vercel.app"
```

你无需在两个配置中重复声明允许的主机。

## 静态基础 URL（向后兼容）

传统的静态字符串配置仍然有效：

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: "https://myapp.com",
})
```

当你的应用仅运行在单一确定的 URL 上时，推荐使用此方式。

## 常见模式

### Vercel 部署

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: [
            "myapp.com",        // 生产自定义域
            "www.myapp.com",    // WWW 变体
            "*.vercel.app",     // 所有预览部署
        ],
    },
})
```

### 开发 + 生产

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: [
            "localhost:3000",
            "localhost:5173",
            "myapp.com",
            "*.vercel.app",
        ],
        protocol: process.env.NODE_ENV === "development" ? "http" : "https",
    },
})
```

### 多生产域名

```ts title="auth.ts"
export const auth = betterAuth({
    baseURL: {
        allowedHosts: [
            "myapp.com",
            "myapp.co.uk",
            "myapp.eu",
        ],
    },
})
```

## 跨子域 Cookie

如果需要在子域之间共享 Cookie，可以开启 `crossSubDomainCookies`，并且不指定静态 `domain`。Better Auth 会自动从每个请求解析的主机推断 Cookie 域。

```ts title="auth.ts"
import { betterAuth } from "better-auth"

export const auth = betterAuth({
    baseURL: {
        allowedHosts: [
            "auth.example1.com",
            "auth.example2.com",
        ],
        protocol: "https",
    },
    advanced: {
        crossSubDomainCookies: {
            enabled: true,
        },
    },
})
```

采用此配置后：

* 对 `auth.example1.com` 的请求会设置 `Domain=auth.example1.com` 的 Cookie
* 对 `auth.example2.com` 的请求会设置 `Domain=auth.example2.com` 的 Cookie

如果需要对所有主机使用相同的域名，可以显式设置 `domain`：

```ts title="auth.ts"
import { betterAuth } from "better-auth"

export const auth = betterAuth({
    baseURL: {
        allowedHosts: ["auth.example1.com", "auth.example2.com"],
        protocol: "https",
    },
    advanced: {
        crossSubDomainCookies: {
            enabled: true,
            domain: ".example.com", // 不论主机均使用此域
        },
    },
})
```

<Callout type="info">
  使用静态的 `baseURL` 字符串时，`crossSubDomainCookies` 在初始化时从该字符串派生域名。使用动态基础 URL 时，域名会根据解析后的主机在每个请求中计算。有关跨子域 Cookie 的更多信息，请参考 [Cookies 文档](/docs/concepts/cookies#cross-subdomain-cookies)。
</Callout>

## 安全注意事项

<Callout type="info">
  动态基础 URL 使用 **基于允许列表的方法** 来保证安全。只有在 `allowedHosts` 中明确列出的主机才会被接受。
</Callout>

1. **允许列表为必填** — 不接受任意主机
2. **头部验证** — 对 `x-forwarded-host` 和 `host` 头进行净化处理后使用
3. **显式模式** — 无自动平台检测；必须手动添加每种模式
4. **明确错误** — 未知主机将抛出描述性错误（除非配置了 fallback）

这与一些自动信任平台或接受任意头部值的其他认证库不同。Better Auth 优先考虑安全性。

## 配置参考

| 属性       | 类型                          | 默认值  | 描述                                                                             |
| ---------- | ----------------------------- | -------- | -------------------------------------------------------------------------------- |
| `allowedHosts` | `string[]`                    | 必需     | 允许的主机模式列表                                                               |
| `fallback`     | `string`                      | -        | 当主机未匹配时使用的 URL                                                         |
| `protocol`     | `"http" \| "https" \| "auto"` | `"auto"` | URL 构造时使用的协议。`"auto"` 从请求头推导，默认为 HTTPS                        |
