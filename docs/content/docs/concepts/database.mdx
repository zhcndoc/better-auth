## 适配器

Better Auth 连接数据库以存储数据。数据库将用于存储用户、会话等数据。插件也可以定义自己的数据库表以存储数据。

你可以通过在数据库选项中传入支持的数据库实例，将数据库连接传递给 Better Auth。你可以在 [其他关系型数据库](/docs/adapters/other-relational-databases) 文档中了解更多支持的数据库适配器。

<Callout type="info">
  Better Auth 同样支持无数据库的使用。更多详情，请参见 [无状态会话管理](/docs/concepts/session-management#stateless-session-management)。
</Callout>

## 命令行工具（CLI）

Better Auth 配备了 CLI 工具，用于管理数据库迁移和生成架构。

### 运行迁移

CLI 会检查你的数据库，并提示你添加缺失的表或用新列更新现有表。这仅支持内置的 Kysely 适配器。对于其他适配器，可以使用 `generate` 命令生成架构，并通过你的 ORM 处理迁移。

```package-install
npx auth@latest migrate
```

<Callout type="info">
  针对 PostgreSQL 用户：migrate 命令支持非默认 schema。它会自动检测你的 `search_path` 配置，并在正确的 schema 中创建表。详细请参见 [PostgreSQL 适配器](/docs/adapters/postgresql#use-a-non-default-schema)。
</Callout>

### 生成架构

Better Auth 还提供了 `generate` 命令，用于生成 Better Auth 所需的架构。若你使用的是 Prisma 或 Drizzle 等数据库适配器，此命令会为你的 ORM 生成正确的架构。如果使用内置的 Kysely 适配器，它将生成一个 SQL 文件，可直接运行于你的数据库。

```package-install
npx auth@latest generate
```

更多 CLI 信息，请参阅 [CLI](/docs/concepts/cli) 文档。

<Callout>
  如果你更喜欢手动添加表也可以。Better Auth 所需的核心架构如下所述，插件文档中可以找到插件所需的额外架构。
</Callout>

### 编程式迁移

在无法使用 CLI 的环境中（例如 Cloudflare Workers，无服务器函数），你可以使用 `better-auth/db/migration` 中的 `getMigrations` 编程运行迁移。

```typescript
import { getMigrations } from "better-auth/db/migration";
import { auth } from "./auth";

const { toBeCreated, toBeAdded, runMigrations } = await getMigrations(auth.options);

await runMigrations();
```

<Callout type="warn">
  `getMigrations` 仅支持内置的 Kysely 适配器（SQLite/D1、PostgreSQL、MySQL、MSSQL）。不支持 Prisma 或 Drizzle ORM 适配器 —— 对于它们，请使用 CLI 迁移。
</Callout>

<Accordions>
  <Accordion id="cloudflare-d1-example" title="示例：Cloudflare D1">
    [Cloudflare D1](https://developers.cloudflare.com/d1/) 只能通过 Cloudflare Worker 查询，因此 CLI 无法直接访问。你可以通过端点运行编程迁移：

    ```typescript title="auth.ts"
    import { env } from "cloudflare:workers";
    import { betterAuth } from "better-auth";

    export const auth = betterAuth({
      database: env.DB,
      // ... 其他配置
    });
    ```

    ```typescript title="src/index.ts"
    import { Hono } from "hono";
    import { auth } from "./auth";
    import { getMigrations } from "better-auth/db/migration";

    const app = new Hono();

    // 生产环境请保护或移除此端点
    app.post("/migrate", async (c) => {
      try {
        const { toBeCreated, toBeAdded, runMigrations } = await getMigrations(auth.options);

        if (toBeCreated.length === 0 && toBeAdded.length === 0) {
          return c.json({ message: "无需迁移" });
        }

        await runMigrations();
        return c.json({
          message: "迁移成功完成",
          created: toBeCreated.map((t) => t.table),
          added: toBeAdded.map((t) => t.table),
        });
      } catch (error) {
        return c.json(
          { error: error instanceof Error ? error.message : "迁移失败" },
          500,
        );
      }
    });

    app.on(["POST", "GET"], "/api/auth/*", (c) => {
      return auth.handler(c.req.raw);
    });

    export default app;
    ```

    <Callout type="info">
      如果你使用 Cloudflare D1 搭配 Drizzle 或 Prisma，使用 [`cloudflare:workers`](https://developers.cloudflare.com/workers/runtime-apis/bindings/) 访问 `env`，并参考以下指南：

      * [Drizzle 与 Cloudflare D1 指南](https://orm.drizzle.team/docs/guides/d1-http-with-drizzle-kit)
      * [Prisma 与 Cloudflare D1 指南](https://www.prisma.io/docs/guides/cloudflare-d1)
    </Callout>

</Accordions>

## 二级存储

Better Auth 的辅助存储允许您使用键值存储来管理会话数据、验证记录、限流计数器以及其他短暂的认证数据。当您希望将高强度记录的存储卸载到高性能存储甚至内存时，这会非常有用。

### 实现

要使用二级存储，实现 `SecondaryStorage` 接口：

```typescript
interface SecondaryStorage {
  get: (key: string) => Promise<unknown>;
  set: (key: string, value: string, ttl?: number) => Promise<void>;
  delete: (key: string) => Promise<void>;
}
```

然后，将你的实现传递给 `betterAuth` 函数：

```typescript title="auth.ts"
import { betterAuth } from "better-auth";

betterAuth({
  // ... 其他选项
  secondaryStorage: {
    // 你的实现
  },
});
```

### Redis 存储

Better Auth 提供官方的 Redis 存储包，基于 [ioredis](https://github.com/redis/ioredis)：

```bash
npm install @better-auth/redis-storage ioredis
# 或者
pnpm add @better-auth/redis-storage ioredis
```

**使用示例：**

```typescript
import { betterAuth } from "better-auth";
import { Redis } from "ioredis";
import { redisStorage } from "@better-auth/redis-storage";

const redis = new Redis({
	host: "localhost",
	port: 6379,
});

export const auth = betterAuth({
	// ... 其他选项
	secondaryStorage: redisStorage({
		client: redis,
		keyPrefix: "better-auth:", // 可选，默认为 "better-auth:"
	}),
});
```

Redis 存储支持 ioredis 的所有连接模式，包括单机、集群和哨兵配置。

**手动实现示例：**

如果你想自己实现 Redis 二级存储，下面是一个基础示例：

```typescript title="auth.ts"
import { createClient } from "redis";
import { betterAuth } from "better-auth";

const redis = createClient();
await redis.connect();

export const auth = betterAuth({
	// ... 其他选项
	secondaryStorage: {
		get: async (key) => {
			return await redis.get(key);
		},
		set: async (key, value, ttl) => {
			if (ttl) await redis.set(key, value, { EX: ttl });
			else await redis.set(key, value);
		},
		delete: async (key) => {
			await redis.del(key);
		}
	}
});
```

此实现允许 Better Auth 使用 Redis 存储会话数据和限速计数器。你还可以为键名添加前缀。

**示例：Upstash Redis 实现**

以下是使用 Upstash Redis 的示例。首先，安装 Upstash Redis 客户端：

```bash
npm install @upstash/redis
```

然后，创建一个新的 Redis 客户端：

```typescript
// src/lib/redis/index.ts

import { Redis } from "@upstash/redis";

export const redis = Redis.fromEnv();
```

别忘了设置 `UPSTASH_REDIS_REST_URL` 和 `UPSTASH_REDIS_REST_TOKEN` 环境变量。有关如何使用 Upstash Redis 结合 Node.js 的更多信息，请参见 [Upstash 文档](https://upstash.com/docs/redis/howto/connectwithupstashredis)。

之后，我们可以创建一个二级存储的实现：

```typescript
// src/lib/auth/adapters/redis-secondary-storage.ts

import { SecondaryStorage } from "better-auth";
import { redis } from "~/lib/redis";

export const redisSecondaryStorage: SecondaryStorage = {
  async get(key: string) {
    try {
      const value = await redis.get(key);

      // 处理 Redis 返回的不同类型
      if (value === null || value === undefined) {
        return null;
      }

      // 如果已经是字符串，则直接返回
      if (typeof value === "string") {
        return value;
      }

      // 如果是对象，则进行字符串化
      if (typeof value === "object") {
        return JSON.stringify(value);
      }

      // 其他类型转换为字符串返回
      return String(value);
    } catch (error) {
      console.error("Redis get 错误:", error);
      return null;
    }
  },

  async set(key: string, value: string, ttl?: number) {
    try {
      // 确保值为字符串
      const stringValue =
        typeof value === "string" ? value : JSON.stringify(value);

      if (ttl) {
        // 带 TTL（秒）设置
        await redis.set(key, stringValue, { ex: ttl });
      } else {
        // 不带 TTL 设置
        await redis.set(key, stringValue);
      }
    } catch (error) {
      console.error("Redis set 错误:", error);
      throw error;
    }
  },

  async delete(key: string) {
    try {
      await redis.del(key);
    } catch (error) {
      console.error("Redis delete 错误:", error);
      throw error;
    }
  },
};
```

最后，我们可以将该实现传递给 `betterAuth` 函数。

```typescript
import { betterAuth } from "better-auth";
import { redisSecondaryStorage } from "~/lib/auth/adapters/redis-secondary-storage";

export const auth = betterAuth({
  // ...其他选项
  secondaryStorage: redisSecondaryStorage,
});
```

## Custom Tables

Better Auth allows you to customize the table names and column names of the core schema. You can also extend the core schema by adding extra fields to the user and session tables.

### Custom Table Names

You can customize the table names and column names of the core schema by using the `modelName` and `fields` properties in your auth configuration:

```ts title="auth.ts"
export const auth = betterAuth({
  user: {
    modelName: "users",
    fields: {
      name: "full_name",
      email: "email_address",
    },
  },
  session: {
    modelName: "user_sessions",
    fields: {
      userId: "user_id",
    },
  },
});
```

<Callout>
  Type inference in your code still uses the original field names (for example, `user.name`, not `user.full_name`).
</Callout>

For plugin-specific table and column names, you can use the `schema` property in the plugin configuration:

```ts title="auth.ts"
import { betterAuth } from "better-auth";
import { twoFactor } from "better-auth/plugins";

export const auth = betterAuth({
  plugins: [
    twoFactor({
      schema: {
        user: {
          fields: {
            twoFactorEnabled: "two_factor_enabled",
            secret: "two_factor_secret",
          },
        },
      },
    }),
  ],
});
```

### Extending the Core Schema

Better Auth provides a type-safe way to extend the `user` and `session` schemas. You can add custom fields in your auth configuration, and the CLI will automatically update the database schema. These extra fields will be correctly inferred in functions that accept user or session objects, such as `useSession` and `signUp.email`.

To add custom fields, use the `additionalFields` property in the `user` or `session` object of your auth configuration. The `additionalFields` object uses the field name as the key, and the value is a `FieldAttributes` object containing:

* `type`: The field data type, such as `"string"`, `"number"`, or `"boolean"`.
* `required`: A boolean indicating whether the field is required.
* `defaultValue`: The default value of the field (note: this only takes effect at the JavaScript layer; the database column itself remains optional).
* `input`: Determines whether the value can be provided when creating a new record (defaults to `true`). If a field like `role` should not be provided by users during sign-up, set this to `false`.

Here is an example of extending the user schema with additional fields:

```ts title="auth.ts"
import { betterAuth } from "better-auth";

export const auth = betterAuth({
  user: {
    additionalFields: {
      role: {
        type: ["user", "admin"],
        required: false,
        defaultValue: "user",
        input: false, // Do not allow users to set the role
      },
      lang: {
        type: "string",
        required: false,
        defaultValue: "en",
      },
    },
  },
});
```

This allows you to access these additional fields in your application logic.

```ts
// During sign-up
const res = await auth.api.signUpEmail({
	body: {
		email: 'test@example.com',
		password: 'password',
		name: 'John Doe',
		lang: 'fr',
	},
});

// user object
res.user.role; // > "admin"
res.user.lang; // > "fr"
```

<Callout>
  For more information on how to infer additional field information on the client side, see the
  [TypeScript](/docs/concepts/typescript#inferring-additional-fields-on-client) documentation.
</Callout>

If you use social/OAuth providers, you may want to provide `mapProfileToUser` to map profile data to the user object so that additional fields can be populated from the provider profile.

**Map profile to user, example: mapping `firstName` and `lastName`**

```ts title="auth.ts"
import { betterAuth } from "better-auth";

export const auth = betterAuth({
  socialProviders: {
    github: {
      clientId: "YOUR_GITHUB_CLIENT_ID",
      clientSecret: "YOUR_GITHUB_CLIENT_SECRET",
      mapProfileToUser: (profile) => {
        return {
          firstName: profile.name.split(" ")[0],
          lastName: profile.name.split(" ")[1],
        };
      },
    },
    google: {
      clientId: "YOUR_GOOGLE_CLIENT_ID",
      clientSecret: "YOUR_GOOGLE_CLIENT_SECRET",
      mapProfileToUser: (profile) => {
        return {
          firstName: profile.given_name,
          lastName: profile.family_name,
        };
      },
    },
  },
});
```

### ID Generation

By default, Better Auth generates unique IDs for users, sessions, and other entities.
You can customize ID generation behavior through the `advanced.database.generateId` option.

#### Option 1: Let the Database Generate IDs

Set `generateId` to `false` to let the database handle all ID generation: (except when `generateId` is `"serial"` and certain `uuid` cases)

```ts title="auth.ts"
import { betterAuth } from "better-auth";
import { db } from "./db";

export const auth = betterAuth({
  database: db,
  advanced: {
    database: {
      generateId: false, // If you want auto-increment numeric IDs, use "serial"
    },
  },
});
```

#### Option 2: Custom ID Generation Function

Use a function to generate IDs. Returning `false` or `undefined` tells the database to generate the ID for a specific model:

```ts title="auth.ts"
import { betterAuth } from "better-auth";
import { db } from "./db";

export const auth = betterAuth({
  database: db,
  advanced: {
    database: {
      generateId: (options) => {
        // Let the database generate IDs for specific models
        if (options.model === "user" || options.model === "users") {
          return false; // Database generates the ID
        }
        // Other tables generate UUIDs
        return crypto.randomUUID();
      },
    },
  },
});
```

<Callout type="info">
  **Important**: When the `generateId` function returns `false` or `undefined`, the corresponding model's ID is generated by the database. Directly setting `generateId: false` (not a function) disables ID generation for **all** tables.
</Callout>

#### Option 3: Unified Custom ID Generator

Generate the same type of ID for all tables:

```ts title="auth.ts"
import { betterAuth } from "better-auth";
import { db } from "./db";

export const auth = betterAuth({
  database: db,
  advanced: {
    database: {
      generateId: () => crypto.randomUUID(),
    },
  },
});
```

### Numeric IDs

If you prefer auto-increment numeric IDs, set `advanced.database.generateId` to `"serial"`.
In this case, Better Auth will not generate IDs and will instead assume the database generates numeric IDs automatically.

When enabled, Better Auth CLI will generate or migrate your database's `id` fields as numeric types with auto-increment properties.

```ts title="auth.ts"
import { betterAuth } from "better-auth";
import { db } from "./db";

export const auth = betterAuth({
  database: db,
  advanced: {
    database: {
      generateId: "serial",
    },
  },
});
```

<Callout type="info">
  Better Auth still treats the `id` field in the database as a string type, and automatically converts it to a numeric type when retrieving or inserting data.

  The `id` you get from Better Auth may be in string form representing a number, which is normal. The `id` you pass to Better Auth should also be a string.
</Callout>

### UUID

If you prefer a UUID type `id` field, set `advanced.database.generateId` to `"uuid"`.
By default, Better Auth generates UUIDs for all tables except the PostgreSQL adapter, which supports database-generated UUIDs.

When enabled, the Better Auth CLI will generate or migrate your database schema with UUID-typed `id` fields.
If the database does not support UUID types, it falls back to plain string types.

### Mixed ID Types

If you need different ID types across tables (for example, integers for users and UUID strings for other tables), use the `generateId` callback function approach.

```ts title="auth.ts"
import { betterAuth } from "better-auth";
import { db } from "./db";

export const auth = betterAuth({
  database: db,
  user: {
    modelName: "users", // PostgreSQL: id is a serial primary key
  },
  session: {
    modelName: "session", // PostgreSQL: id is a text primary key
  },
  advanced: {
    database: {
      // Do not set useNumberId — this option is global and affects all tables
      generateId: (options) => {
        if (options.model === "user" || options.model === "users") {
          return false; // Let PostgreSQL serial generate the ID
        }
        return crypto.randomUUID(); // session, account, verification generate UUIDs
      },
    },
  },
});
```

This configuration supports:

* The user table uses database auto-increment (serial, auto\_increment, etc.)
* Other tables (session, account, verification) generate UUIDs
* Compatibility with existing schemas that use different ID types

<Callout type="info">
  **Use case**: This is especially useful when migrating from another auth provider (such as Clerk), where your user IDs may be integers and new tables use UUID strings.
</Callout>

### Database Hooks

Database hooks allow you to define custom logic that runs during Better Auth's core database operation lifecycle. You can create hooks for the following models: **user**, **session**, and **account**.

<Callout type="warn">
  Additional fields are supported, but full type inference for these fields is not yet supported and will be improved in the future.
</Callout>

You can define two types of hooks:

#### 1. Before Hook

* **Purpose**: Called before the corresponding entity (user, session, or account) is created, updated, or deleted.
* **Behavior**: Returning `false` aborts the operation, while returning a data object replaces the original payload.

#### 2. After Hook

* **Purpose**: Called after the corresponding entity is successfully created or updated.
* **Behavior**: Can perform additional actions or modifications.

**Example Usage**

```typescript title="auth.ts"
import { betterAuth } from "better-auth";

export const auth = betterAuth({
  databaseHooks: {
    user: {
      create: {
        before: async (user, ctx) => {
          // Modify the user object before creation
          return {
            data: {
              // Note: return Better Auth named fields, not the raw database field names
              ...user,
              firstName: user.name.split(" ")[0],
              lastName: user.name.split(" ")[1],
            },
          };
        },
        after: async (user) => {
          // Perform additional actions, such as creating a Stripe customer
        },
      },
      delete: {
        before: async (user, ctx) => {
          console.log(`User ${user.email} is being deleted`);
          if (user.email.includes("admin")) {
            return false; // Abort deletion
          }
          return true; // Allow deletion
        },
        after: async (user) => {
          console.log(`User ${user.email} has been deleted`);
        },
      },
    },
    session: {
      delete: {
        before: async (session, ctx) => {
          console.log(`Session ${session.token} is being deleted`);
          if (session.userId === "admin-user-id") {
            return false; // Abort deletion
          }
          return true; // Allow deletion
        },
        after: async (session) => {
          console.log(`Session ${session.token} has been deleted`);
        },
      },
    },
  },
});
```

#### Throwing Errors

If you want to stop execution in a database hook, you can throw an error using the `APIError` class imported from `better-auth/api`.

```typescript title="auth.ts"
import { betterAuth } from "better-auth";
import { APIError } from "better-auth/api";

export const auth = betterAuth({
  databaseHooks: {
    user: {
      create: {
        before: async (user, ctx) => {
          if (user.isAgreedToTerms === false) {
            // Your custom condition
            // Throw an API error
            throw new APIError("BAD_REQUEST", {
              message: "Users must agree to the terms of service before registering.",
            });
          }
          return {
            data: user,
          };
        },
      },
    },
  },
});
```

#### Using the Context Object

The context object (`ctx`) is passed as the second parameter to hooks and contains useful information. For `update` hooks, `ctx` includes the current `session`, allowing access to the authenticated user's details.

```typescript title="auth.ts"
import { betterAuth } from "better-auth";

export const auth = betterAuth({
  databaseHooks: {
    user: {
      update: {
        before: async (data, ctx) => {
          // Access session through ctx
          if (ctx.context.session) {
            console.log("User update initiated by:", ctx.context.session.userId);
          }
          return { data };
        },
      },
    },
  },
});
```

Database hooks, like standard hooks, also provide rich `ctx` object properties. See the [hooks documentation](/docs/concepts/hooks#ctx) for details.

## Plugin Schema

Plugins can define their own tables in the database to store more data, and can also add fields to core tables to store additional information. For example, the two-factor authentication plugin adds the following fields to the `user` table:

* `twoFactorEnabled`: Whether the user has two-factor authentication enabled.
* `twoFactorSecret`: The secret used to generate TOTP codes.
* `twoFactorBackupCodes`: Encrypted backup codes for account recovery.

You have two options for adding tables and columns to the database:

* **CLI**: Use the migrate or generate command. The command scans the database and guides you in adding missing tables or columns.
* **Manual method**: Follow the plugin documentation to add tables and columns manually.

Both approaches ensure that the database schema stays in sync with plugin requirements.

## Experimental Joins

Starting from Better Auth `1.4`, experimental database join support has been added.
This allows Better Auth to execute multiple database queries in a single request, reducing database round trips.
More than 50 endpoints support joins, and we are continuously adding more.

Internally, our adapter system natively supports joins. Even if you do not enable experimental joins,
it will still fall back to executing multiple queries and merging the results.

To enable joins, add the following to your auth configuration:

```ts title="auth.ts"
import { betterAuth } from "better-auth";

export const auth = betterAuth({
  experimental: { joins: true }
});
```

Better Auth `1.4` CLI automatically generates relationships for DrizzleORM and PrismaORM, so if you have not added them yet, run the migrate or generate command to update to the latest required schema.

We strongly recommend reading the experimental joins documentation for the corresponding adapter:

* [DrizzleORM](/docs/adapters/drizzle#joins-experimental)
* [PrismaORM](/docs/adapters/prisma#joins-experimental)
* [SQLite](/docs/adapters/sqlite#joins-experimental)
* [MySQL](/docs/adapters/mysql#joins-experimental)
* [PostgreSQL](/docs/adapters/postgresql#joins-experimental)
* [MSSQL](/docs/adapters/mssql#joins-experimental)
* [MongoDB](/docs/adapters/mongo#joins-experimental)
