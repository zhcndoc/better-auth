本指南将带您了解如何将项目从 WorkOS 迁移到 Better Auth，涵盖如何迁移一个与 Next.js 应用集成的基础 WorkOS 设置及关键注意事项。

## 开始前

开始之前，让我们回顾一下哪些 WorkOS 认证功能在 Better Auth 中得到了完全或部分支持。如果您在 WorkOS 中使用的某个功能是通过插件提供的，您需要在下一步进行配置。

## 创建 Better Auth 实例

首先，在项目中设置 Better Auth。可参考[安装指南](/docs/installation)开始。

### 数据库

Better Auth 支持多种数据库，设置您偏好的数据库。在本指南中，我们将使用 PostgreSQL 及默认数据库适配器。

```ts title="auth.ts"
import { betterAuth } from "better-auth";
import { Pool } from "pg";

export const auth = betterAuth({
  database: new Pool({ // [!code highlight]
    connectionString: process.env.DATABASE_URL, // [!code highlight]
  }), // [!code highlight]
});
```

### 邮箱 & 密码

启用邮箱 + 密码认证，示例如下。由于 WorkOS 默认验证每个用户邮箱，此设置与默认行为相似，您可视需求调整。详情见[这里](/docs/authentication/email-password)。

```ts
import { betterAuth } from "better-auth";
import { Pool } from "pg";

export const auth = betterAuth({
  database: new Pool({
    connectionString: process.env.DATABASE_URL,
  }),
  emailAndPassword: { // [!code highlight]
    enabled: true, // [!code highlight]
    requireEmailVerification: true, // [!code highlight]
    minPasswordLength: 10, // [!code highlight]
    sendResetPassword: async ({ user, url, token }, request) => { // [!code highlight]
      // 实现您的邮件发送逻辑 // [!code highlight]
    }, // [!code highlight]
  }, // [!code highlight]
  emailVerification: { // [!code highlight]
    sendVerificationEmail: async ({ user, url, token }, request) => { // [!code highlight]
      // 实现您的邮件发送逻辑 // [!code highlight]
    }, // [!code highlight]
  }, // [!code highlight]
});
```

### 社交提供商（可选）

按以下配置您在 WorkOS 中使用的社交提供商。Better Auth 支持更多提供商，您也可以添加更多。由于 WorkOS 确保邮箱唯一，请在 Better Auth 中配置 `account.accountLinking` 以保证同样行为。

```ts
import { betterAuth } from "better-auth";
import { Pool } from "pg";

export const auth = betterAuth({
  // ... 其他选项

  socialProviders: { // [!code highlight]
    github: { // [!code highlight]
      clientId: process.env.GITHUB_CLIENT_ID!, // [!code highlight]
      clientSecret: process.env.GITHUB_CLIENT_SECRET!, // [!code highlight]
    }, // [!code highlight]
    // ... 其他提供商 // [!code highlight]
  }, // [!code highlight]
  account: { // [!code highlight]
    accountLinking: { // [!code highlight]
      enabled: true, // [!code highlight]
      trustedProviders: ["email-password", "github"], // [!code highlight]
    }, // [!code highlight]
  }, // [!code highlight]
});
```

### 插件

请参阅[映射部分](#开始前)中 WorkOS 功能到 Better Auth 的对应关系。如果您在 WorkOS 中使用的功能在 Better Auth 中有对应插件，需将其添加到插件选项中。Better Auth 通过插件提供了更多开箱即用的功能。详情见[这里](/docs/concepts/plugins)。

```ts title="auth.ts"
  ], // [!code highlight]
});
```

## 生成数据库模式

Better Auth 允许您完全控制数据库，且可以通过 CLI 轻松生成认证实例的数据库模式。详情见[这里](/docs/concepts/cli)。

### 默认数据库适配器

运行 `migrate` 命令，在数据库中创建 Better Auth 实例所需的模式。

```package-install

```

如果您使用 Prisma、Drizzle 等数据库适配器，可使用 `generate` 命令生成 ORM 的模式文件。然后通过外部工具（如 Drizzle Kit）执行迁移。


## 迁移脚本

### 创建迁移脚本

编写迁移脚本，将您的用户数据从 WorkOS 导入到数据库。

```ts title="scripts/migration.ts"
import { auth } from "@/lib/auth"; // 您的 auth 实例路径
import { WorkOS } from "@workos-inc/node";

//==============================================================================

/*
  速率限制配置

  WorkOS 读取 API：每 10 秒最多 1,000 次请求
  默认设置：使用限制的 80% 避免极端情况

  参考：https://workos.com/docs/reference/rate-limits
*/
const TIME_WINDOW_MS = 10 * 1000; // 时间窗口，单位毫秒（10 秒）
const MAX_REQUESTS_PER_WINDOW = 800; // 时间窗口内最大 API 调用次数
const USERS_PER_REQUEST = 100; // 每次 API 调用获取的用户数

//==============================================================================

if (!process.env.WORKOS_API_KEY || !process.env.WORKOS_CLIENT_ID) {
  throw new Error(
    "缺少必需的环境变量 WORKOS_API_KEY 及/或 WORKOS_CLIENT_ID",
  );
}
const workos = new WorkOS(process.env.WORKOS_API_KEY);

/**
 * 创建速率限制器来跟踪并控制请求速率
 */
const createRateLimiter = (maxRequests: number, windowMs: number) => {
  let requestTimestamps: number[] = [];

  const waitIfNeeded = async (): Promise<void> => {
    const now = Date.now();

    // 删除当前窗口之外的时间戳
    requestTimestamps = requestTimestamps.filter(
      (timestamp) => now - timestamp < windowMs,
    );

    // 达到限制时计算等待时间
    if (requestTimestamps.length >= maxRequests) {
      const oldestTimestamp = requestTimestamps[0]!;
      const waitTime = windowMs - (now - oldestTimestamp) + 1000; // 留 1 秒缓冲

      console.log(
        `⏳ 速率限制中 (${requestTimestamps.length}/${maxRequests} 调用已用)，等待 ${Math.ceil(waitTime / 1000)} 秒...`,
      );
      await new Promise((resolve) => setTimeout(resolve, waitTime));

      // 等待后清理旧时间戳
      const newNow = Date.now();
      requestTimestamps = requestTimestamps.filter(
        (timestamp) => newNow - timestamp < windowMs,
      );
    }

    // 记录此次请求
    requestTimestamps.push(Date.now());
  };

  const getStats = (): {
    current: number;
    max: number;
    windowMinutes: number;
  } => {
    const now = Date.now();
    requestTimestamps = requestTimestamps.filter(
      (timestamp) => now - timestamp < windowMs,
    );

    return {
      current: requestTimestamps.length,
      max: maxRequests,
      windowMinutes: windowMs / (60 * 1000),
    };
  };

  return { waitIfNeeded, getStats };
};

/**
 * 安全地将各种日期格式转换为 Date 对象。
 * 转换失败时返回当前日期（适用于 createdAt/updatedAt）。
 */
const safeDateConversion = (date?: string | number | Date | null): Date => {
  if (date == null) return new Date();

  if (date instanceof Date) return new Date(date.getTime());

  if (typeof date === "number") {
    if (!Number.isFinite(date)) return new Date();
    return new Date(date);
  }

  if (typeof date === "string") {
    const trimmed = date.trim();
    if (trimmed === "") return new Date();
    const parsed = new Date(trimmed);
    if (isNaN(parsed.getTime())) return new Date();
    return parsed;
  }

  return new Date();
};

/**
 * 安全地将 firstName 和 lastName 转换为全名字符串。
 * 若两者均为空，返回 "Username"。
 */
const safeNameConversion = (
  firstName?: string | null,
  lastName?: string | null,
): string => {
  const trimmedFirstName = firstName?.trim();
  const trimmedLastName = lastName?.trim();

  if (trimmedFirstName && trimmedLastName) {
    return `${trimmedFirstName} ${trimmedLastName}`;
  }

  if (trimmedFirstName) return trimmedFirstName;
  if (trimmedLastName) return trimmedLastName;

  return "Username";
};

async function migrateFromWorkOS() {
  const ctx = await auth.$context;
  const rateLimiter = createRateLimiter(
    MAX_REQUESTS_PER_WINDOW,
    TIME_WINDOW_MS,
  );

  let totalUsers = 0;
  let migratedUsers = 0;
  let skippedUsers = 0;
  let failedUsers = 0;

  let hasMoreUsers = true;
  let after: string | undefined;
  let batchCount = 0;

  console.log("");
  console.log("=".repeat(40));
  console.log("🚀 开始迁移");
  console.log("");
  console.log(`设置：`);
  console.log(
    ` - 最大 API 调用：${MAX_REQUESTS_PER_WINDOW} 次 / ${TIME_WINDOW_MS / 1000} 秒`,
  );
  console.log(` - 每次调用用户数：${USERS_PER_REQUEST}`);
  console.log("=".repeat(40));
  console.log("");

  while (hasMoreUsers) {
    try {
      await rateLimiter.waitIfNeeded();

      const workosUserList = await workos.userManagement.listUsers({
        limit: USERS_PER_REQUEST,
        after,
      });

      batchCount++;
      console.log(
        `📦 批次 ${batchCount}：从 WorkOS 获取了 ${workosUserList.data.length} 个用户`,
      );

      after = workosUserList.listMetadata.after || undefined;
      hasMoreUsers = !!after;
      totalUsers += workosUserList.data.length;

      for (const workosUser of workosUserList.data) {
        try {
          console.log(`\n处理用户：${workosUser.email}`);

          // 通过邮箱检查用户是否已存在
          // WorkOS 通过邮箱验证流程保证邮箱唯一
          const existingUser = await ctx.adapter.findOne<
            typeof auth.$Infer.Session.user
          >({
            model: "user",
            where: [
              {
                field: "email",
                value: workosUser.email,
              },
            ],
          });

          if (existingUser) {
            console.log(
              `🟡 用户已存在，跳过：${workosUser.email}`,
            );
            skippedUsers++;
            continue;
          }

          // 创建新用户
          await ctx.adapter.create<typeof auth.$Infer.Session.user>({
            model: "user",
            data: {
              email: workosUser.email,
              emailVerified: workosUser.emailVerified,
              image: workosUser.profilePictureUrl,
              name: safeNameConversion(
                workosUser.firstName,
                workosUser.lastName,
              ),
              createdAt: safeDateConversion(workosUser.createdAt),
              updatedAt: safeDateConversion(workosUser.updatedAt),
              metadata: {
                workosId: workosUser.id,
                ...(workosUser.metadata || {}),
              },
            },
          });

          console.log(`🟢 迁移用户成功 ${workosUser.email}`);
          migratedUsers++;
        } catch (error) {
          console.error(
            `🔴 迁移用户失败 ${workosUser.email}\n`,
            error,
          );
          failedUsers++;
        }
      }

      console.log("");
    } catch (error) {
      console.error("🚨 获取批次数据时出错:", error);
      throw error;
    }
  }

  console.log("");
  console.log("=".repeat(40));
  console.log("📝 迁移总结");
  console.log(`处理用户总数：${totalUsers}`);
  console.log("");
  console.log(`🔴 失败数：${failedUsers}`);
  console.log(`🟡 跳过数：${skippedUsers}`);
  console.log(`🟢 成功迁移数：${migratedUsers}`);
  console.log("=".repeat(40));
}

async function main() {
  try {
    await migrateFromWorkOS();
    process.exit(0);
  } catch (error) {
    console.error("\n迁移失败:", error);
    process.exit(1);
  }
}
main();
```

<Callout type="info">
**注意**

- 从 WorkOS 获取用户数据需要调用其 API，API 有速率限制。示例脚本包含基础配置，请根据您的环境调整。
- 本迁移脚本涵盖了邮箱+密码和社交登录常见用户管理情况。对于 SSO 或 CLI 认证等 Better Auth 插件功能，请根据示例调整脚本。
</Callout>

## 资源保护

> Proxy（中间件）不适合慢速数据获取。虽然 Proxy 可以用于权限跳转等乐观检查，但不应作为完整会话管理或授权方案。 - [Next.js 文档](https://nextjs.org/docs/app/getting-started/proxy#use-cases)

### 中间件认证

WorkOS 提供 Proxy（中间件）认证。Better Auth 不推荐直接在中间件中保护资源，因此不提供专用辅助函数。

```ts title="proxy.ts / middleware.ts"
import { authkitMiddleware } from '@workos-inc/authkit-nextjs';

export default authkitMiddleware({
  middlewareAuth: {
    enabled: true,
    unauthenticatedPaths: ['/'],
  },
});

export const config = { matcher: ['/', '/account/:page*'] };
```

在 Better Auth 中，为了方便而非资源保护，中间件代理可按以下方式使用。该方案支持 Next.js 15+ 及 Node.js 运行时。

```ts title="proxy.ts"
import { NextRequest, NextResponse } from "next/server";
import { headers } from "next/headers";
import { auth } from "@/lib/auth";

export async function proxy(request: NextRequest) {
    const session = await auth.api.getSession({
        headers: await headers()
    })

    // 推荐乐观跳转用户的处理方案
    // 建议在每个页面/路由中处理认证检查
    if(!session) {
        return NextResponse.redirect(new URL("/sign-in", request.url));
    }

    return NextResponse.next();
}

export const config = {
  matcher: ["/dashboard"], // 指定中间件应用的路由
};
```

## 注意事项

<strong className='underline italic'>密码哈希</strong>

如果您管理的用户使用邮箱 + 密码系统，则 WorkOS 目前不支持导出密码哈希。迁移后，用户需要在您的认证系统中重置密码。请务必在迁移前后充足时间内通知用户此变更。

<strong className='underline italic'>数据同步</strong>

WorkOS 作为托管服务，通过 API 或 Webhook 与您的服务器保持数据同步。Better Auth 则由您自行完全掌控认证系统，通过 API 自由管理数据。但如果之前依赖 Webhook 实现同步，则需要额外调整。

<strong className='underline italic'>停机时间</strong>

WorkOS 通过其 API 暴露数据，但有限制，如不支持导出密码哈希。基于这些限制，实现零停机迁移较为困难。请规划好迁移流程，预留足够缓冲时间，并告知用户预期影响。

<strong className='underline italic'>活跃会话</strong>

现有活跃会话不会被迁移。迁移后用户需重新登录，务必提前通知用户。

## 总结

恭喜！您已成功完成从 WorkOS 迁移至 Better Auth。Better Auth 提供了更高的灵活性和更多功能，务必深入探索[文档](/docs)以发挥其全部潜力。
