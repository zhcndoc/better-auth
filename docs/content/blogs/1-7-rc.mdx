# Better Auth 1.7 RC

我们在 Better Auth 1.7 上投入了很多。这是一个重大版本，其中一部分意味着需要进行迁移工作。我们希望支持更广泛的登录流程，让系统更稳健，并推出那些需要结构性改动的安全修复。

以下是此版本带来的内容。

**你的应用可以成为其他应用的登录提供方。** 如果其他应用通过你的应用登录，Better Auth 现在可以处理更多 OAuth 和 OpenID Connect 规则。这包括 DPoP token、单点登出、强制重新登录、按 API 的 token 规则，以及更安全的 token 检查。

**更多登录提供方可用。** Better Auth 现在支持此前表现不佳的登录配置：Amazon Cognito、使用证书登录的 Microsoft Entra ID、标准 OpenID 提供方，以及像 Clever 这样的学校登录提供方。

**默认情况下更安全。** 我们加强了此前不安全的流程。随着安全性越来越成为重点，我们宁愿披露安全漏洞并迅速修复，也不愿为了向后兼容而继续保留它们。

## 亮点

<HeaderLabel variant="warning">重大变更</HeaderLabel>

### Better Auth 作为身份提供者

这是本次发布中最重要的部分。OAuth 提供程序从“只负责发放 token”成长为一个基于标准的授权服务器，其他应用可以通过它登录。

* **受保护资源：** 将登录服务器后面的每个 API 描述为其自己的资源，拥有各自的 token 生命周期、作用域和声明。token 会被锁定到它所签发给的 API，不能再被重放到其他 API。这取代了扁平的 `validAudiences` 列表。
* **DPoP 绑定 token：** 访问 token 会绑定到客户端持有的密钥，因此即使 token 泄露，也不足以调用 API。
* **单点登出：** OIDC Back-Channel Logout 会在用户从你的提供者登出时，通知所有已登录的应用结束它们各自的会话。
* **强制重新登录：** `max_age` 现在会被正确遵守，而不再被忽略。
* **一致的 introspection：** 不透明 token 返回的声明与 JWT 相同，且独立的 API 服务可以 introspect 为另一个客户端签发的 token。
* **协议安全的 ID token：** 自定义声明不再能够覆盖诸如发行者、主体和受众等保留的协议声明。
* **自助和机器客户端：** 后端可以在没有已登录用户的情况下使用预共享 token 完成注册，而 **Client ID Metadata Documents**（`@better-auth/cimd`）允许客户端通过一个托管 URL 来标识自己，这也是 MCP 客户端无需预先注册即可连接的方式。像 Claude 和 Codex 这样的真实 MCP 客户端，现在默认会保留它们注册时使用的认证方式为私密，而不是被强制公开。
* **请求特定的用户详情：** 客户端可以通过标准的 `claims.userinfo` 参数请求单独的用户声明，并且该请求会成为用户同意的一部分。
* **扩展面：** `extendOAuthProvider` 允许配套插件在不修改核心的情况下添加授权类型、客户端认证方法、发现元数据和声明。
* **一致性打磨：** 标准的 `{ error, error_description }` 封装、凭据响应上的 `no-store`、ID token 上的 `at_hash` 声明、表单编码请求，以及面向原生客户端的刷新重试窗口。

旧的 `oidcProvider` 插件已被移除，它在 1.6 中已被弃用。请将 OpenID provider 配置迁移到 `@better-auth/oauth-provider`。

### 连接到更多身份提供商

OAuth 工作的另一半，是关于 Better Auth 作为接收登录的应用。

* **通用 OAuth 已重构**，与内置社交提供商走同一路径，默认启用 PKCE，并通过发现机制自动进行发行者验证。你调用的 API 和回调 URL 都会改变。
* **身份令牌会一致地校验**，并使用提供商发布的密钥，包括移动端和单页应用。自定义提供商从 `verifyIdToken` 方法迁移到 `idToken` 配置。
* **已授权的范围会被保留**，在重新登录和令牌刷新时不会被覆盖，而 Google 的 `includeGrantedScopes` 现在也可以配置了（默认仍开启）。
* **Google One Tap** 会更严格地验证身份令牌，拒绝格式错误的令牌，并遵循 `requireEmailVerification`。
* **每次请求的登录选项**（`additionalParams`）支持启用 Cognito 上游路由、Microsoft Entra ID 的 `domain_hint`，以及 Google 的离线或增量访问。**证书和签名断言登录**（`clientAssertion`、`tokenEndpointAuth`）支持 Entra ID 证书登录和通用的 `private_key_jwt`。
* **由提供商发起的登录** 可通过 `allowIdpInitiated: true` 安全地重新开始，**按提供商的邮箱验证**（`requireEmailVerification`）会在邮箱验证完成前暂不建立会话，而 **匿名账户关联** 现在可在 Expo 和其他应用内浏览器中正常工作。

### 新增可用的集成

这些配置在以前是不可能实现、会出错或不安全的，现在已经可以正常工作。

**使用外部提供商登录你的用户**

| Provider or setup                  | What was blocked              | How it works now                                      |
| ---------------------------------- | ----------------------------- | ----------------------------------------------------- |
| **Amazon Cognito** 联合登录            | 你无法在每次登录时将用户路由到特定的上游提供商。      | 使用类型化的 `identityProvider` 和按请求传入的 `additionalParams`。 |
| **Microsoft Entra ID** 证书登录        | 仅支持共享密钥登录。                    | 使用 `clientAssertion` 进行证书登录，并在每次登录时使用 `domain_hint`。  |
| **Google** 离线和增量访问                 | 选项是全局性的，并且每次登录都会覆盖已授予的 scope。 | 使用按请求配置的选项、保留的 scopes 和 `includeGrantedScopes`。       |
| **通过发现机制接入的任何 OpenID 提供商**         | 未验证提供商的身份令牌。                  | 指向一个 discovery URL，令牌会自动验证。                           |
| **Zitadel、Auth0 以及其他多租户 OIDC 提供商** | 刷新令牌时无法发送额外参数。                | 刷新时使用 `refreshTokenParams`，无需完整重定向。                   |
| **需要签名断言登录的提供商**                   | 仅支持基于密钥的登录。                   | 使用带签名 JWT 的 `tokenEndpointAuth`。                      |
| **Clever 以及类似的教育类提供商**             | 由提供商发起的登录会失败。                 | 使用 `allowIdpInitiated: true` 安全地重新启动流程。               |

**其他应用通过你进行登录**

| Setup                                   | What was blocked        | How it works now                          |
| --------------------------------------- | ----------------------- | ----------------------------------------- |
| **需要防盗用令牌的 API**                        | 只有普通 bearer token。      | 使用与客户端密钥绑定的 DPoP token。                   |
| **一个登录服务器后面的多个 API**                    | 一个 token 可以在任何 API 上使用。 | 使用按 API 划分的资源，因此 token 只会绑定到它被签发给的 API。   |
| **自行注册的机器客户端**                          | 注册必须依赖已登录用户。            | 使用预共享的注册 token。                           |
| **MCP 客户端（Claude、Codex、Factory Droid）** | 它们的注册会被拒绝。              | 它们会被自动接受，并支持 Client ID Metadata Document。 |
| **跨应用单点登出**                             | 登出不会通知其他应用。             | 使用 OIDC Back-Channel Logout。              |
| **强制重新登录**                              | 请求会被忽略。                 | `max_age` 会被强制执行。                         |

**企业级 SSO**

| Setup                                | What was blocked | How it works now |
| ------------------------------------ | ---------------- | ---------------- |
| **SAML 证书轮换**                        | 只接受一个证书。         | 轮换期间会接受证书列表。     |
| **SCIM 组供应**                         | 没有持久化的组资源。       | 组拥有一级生命周期端点。     |
| **Cloudflare Workers 上的 OpenID SSO** | 重定向端点会破坏流程。      | 它们会以清晰的配置错误失败。   |

### 企业 SSO、SAML 和 SCIM

* **无停机轮换 SAML 证书：** 签名证书现在可以是一个列表，因此管理员可以在旧证书旁边发布新证书。管理端点会以列表形式返回证书，或者在证书位于 `idpMetadata` 文档中时省略它。
* **默认关闭未请求的 SAML 登录：** `allowIdpInitiated` 现在默认值为 `false`，因此除非通过 `allowIdpInitiated: true` 显式启用，否则由身份提供方发起的登录会被拒绝。
* **更简单的 SAML 配置：** 回调 URL 现在会自动推导，服务提供方元数据会自动生成，若干未使用的字段被移除，并且单点注销会正确结束会话。某个端点路径也发生了变化。
* **SCIM 组和组织作用域：** SCIM 新增了持久化的组资源，并且现在每个连接都必须通过必填的 `organizationId` 绑定到某个组织，旧的按用户所有权选项已被移除。这需要一次迁移，并且需要对 1.7 之前且没有组织的连接手动重新认领一次。

<HeaderLabel variant="info">新增</HeaderLabel>

### Drizzle 关系 v2

Drizzle ORM v1 现已进入候选发布版，并带来了全新的 relations API。Better Auth 通过一个新的适配器入口点支持它，因此生成的 auth 关系可以与你应用自身的关系并排使用。

```ts title="auth.ts"
import { betterAuth } from "better-auth/minimal";
import { drizzleAdapter } from "@better-auth/drizzle-adapter/relations-v2"; // [!code highlight]
import { db } from "./db";
import * as schema from "./schema";

export const auth = betterAuth({
  database: drizzleAdapter(db, { provider: "pg", schema }),
});
```

<HeaderLabel variant="warning">重大变更</HeaderLabel>

### 默认安全

* **默认不信任转发代理头：** 你的应用从 `Host` 头读取自身地址，并忽略 `x-forwarded-*`，除非你通过 `advanced.trustedProxyHeaders: true` 显式启用。nginx、Vercel、Cloudflare 和 Netlify 等平台通常无需更改。
* **原子状态，不再有竞争条件：** 数据库中的“先检查，再写入”逻辑被单个原子操作取代，因此一次性令牌不能被重复使用，团队不会超过其限额，限流在高负载下也能保持有效。自定义适配器和存储后端现在需要实现必要的方法。
* **Electron 需要现代 PKCE：** Electron 流程现在要求使用 S256 PKCE，并且不再信任自定义 origin 头。请同时升级客户端和服务器。
* **验证码匹配完整路径：** 将诸如 `/sign-in` 这样的部分路径替换为 `/sign-in/*` 或 `/sign-in/**`。
* **更严格的自定义协议源：** 带主机的 `trustedOrigins` 条目，例如 `myapp://callback`，现在只精确匹配该主机，不再接受 `myapp://callback.attacker.tld`。
* **无密码登录会清除未经证实的凭据：** 当账户邮箱从未被确认时，已验证的邮箱控制权优先，因此在登录前会移除未经证实的密码以及任何其他已关联账户。
* **新身份的门禁：** 新的 `user.validateUserInfo` 钩子可以在创建用户或链接账户之前拒绝某个身份，适用于每一种注册方式。

## 重要变更

### 重大变更

| 变更                                      | 该怎么做                                                                                                                  |
| --------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| 受保护资源取代 `validAudiences`                | 将每个 audience 移到 `resources` 中，将客户端关联到资源，并运行模式迁移。                                                                      |
| DPoP 更改令牌验证                             | 将 bearer-token 辅助函数重命名，在需要的地方使用 DPoP 请求验证器，并运行模式迁移。                                                                   |
| 后通道注销会吊销与会话绑定的令牌                        | 运行模式迁移，并预期与已登出会话绑定的令牌变为无效。                                                                                            |
| 通用 OAuth 使用社交提供方路径                      | 更新登录调用、关联调用、回调 URL 和客户端插件。                                                                                            |
| 自定义社交提供方使用单一身份令牌验证器                     | 将提供方的 `verifyIdToken` 方法替换为 `idToken` 配置。                                                                             |
| 旧的 `oidcProvider` 插件已移除                 | 将提供方设置迁移到 `@better-auth/oauth-provider`。                                                                              |
| MCP 迁移到 `@better-auth/mcp`              | 更新导入、端点路径、辅助函数名、配置结构和模式。                                                                                              |
| SAML 默认值和配置已更改                          | 更新已移除的字段，更新回调 URL，并检查 IdP 发起的流程。                                                                                      |
| SCIM 连接需要手动回收                           | 在重新生成其令牌之前，为 1.7 之前的连接分配 `organizationId`，或删除无主行。                                                                     |
| 默认情况下不信任代理头                             | 仅当你的代理需要时，才通过 `advanced.trustedProxyHeaders: true` 显式启用。                                                              |
| 自定义适配器和存储需要原子方法                         | 实现 `incrementOne` 和 `consumeOne`（适配器），`increment` 和 `getAndDelete`（二级存储），或 `consume`（限流存储）。                           |
| 验证码规则匹配完整路径                             | 将 `/sign-in` 之类的部分路径替换为 `/sign-in/*` 或 `/sign-in/**`。                                                                 |
| 自定义 scheme 的受信任来源按主机匹配                  | 像 `myapp://callback` 这样的带主机条目不再接受 `myapp://callback.attacker.tld`。请重新检查原生和移动端的 `trustedOrigins`。                      |
| Electron 需要 S256 PKCE                   | 一起升级 Electron 客户端和服务端。                                                                                                |
| OIDC ID 令牌不再包含 profile/email scope 声明   | 从 UserInfo 端点读取 profile 和 email 声明，而不是从 ID 令牌中读取。                                                                     |
| 移除了同步 OAuth2 请求构建器                      | 用异步等价方法替换 `createAuthorizationCodeRequest`、`createRefreshAccessTokenRequest` 和 `createClientCredentialsTokenRequest`。 |
| `jwt.sign` 回调必须与 `keyPairConfig.alg` 匹配 | 将你自定义的 ID 令牌签名算法与配置的密钥对保持一致，否则会拒绝签发。                                                                                  |
| 动态客户端注册验证更严格                            | 发送相互对应的 `response_types`/`grant_types`（`code` 响应类型需要 `authorization_code` 授权）。                                        |
| 未认证注册会保留客户端认证方式                         | 默认情况下，注册是机密型的。对于必须保持为公有的客户端，请设置 `token_endpoint_auth_method: "none"`。                                                 |
| `/oauth2/revoke` 拒绝有效的 JWT 访问令牌         | 预期返回 `400 unsupported_token_type`。请改为吊销 refresh token 或不透明令牌。                                                         |
| OAuth 回调错误码重命名                          | 将 `email_doesn't_match` 的处理更新为 `email_does_not_match`。                                                                |
| `generateState()` 签名已更改                 | 改为使用新的选项对象调用它，而不是位置参数 `(c, link, additionalData)`。                                                                    |
| SCIM 连接按组织作用域划分                         | `organizationId` 变为必需，`userId` 被移除，`defaultSCIM` 变为 `staticProviders`，`trustedDomains` 被移除。运行迁移。                      |
| SCIM 需要组织插件                             | 没有它时 SCIM 不再初始化，并且每个令牌都必须绑定到一个组织。                                                                                     |
| SCIM 账户 ID 按组织命名空间隔离                    | 将现有的 SCIM 关联账户迁移为 `scim:{organizationId}:{providerId}` 形式的 provider-id。                                               |
| SSO SAML 配置注册变更                         | 提供签名证书来源，并期望完整的小写 ACS 错误重定向代码，而不是短别名。                                                                                 |
| `/sso/update-provider` 拒绝部分映射           | 发送完整的 OIDC/SAML 映射对象，而不是部分对象。                                                                                         |
| auth CLI 需要 Node.js 22.12+              | 升级用于运行 CLI 的 Node.js。                                                                                                 |
| 新的组织列                                   | 运行 `team.memberCount` 和 `teamMember.membershipKey` 的迁移。                                                               |
| Stripe 组织订阅需要 `organization.enabled`    | 除非在 Stripe 插件配置中设置 `organization: { enabled: true }`，否则 `referenceMiddleware` 会拒绝组织作用域订阅。                             |
| 默认的 Drizzle 模式使用单数关系键                   | 重新生成并检查你的 Drizzle 模式关系。                                                                                               |
| 需要 Stripe `onSubscriptionCancel` 事件     | 更新回调以期望一个非可选的 `event`。                                                                                                |
| 仅支持双因素 OTP 启用                           | `enableTwoFactor` 接受一个新的 `method` 参数并返回一个可辨别联合类型响应。更新调用方。                                                             |
| 公共导出 `getIp` 已重命名为 `getIP`              | 更新 IP 辅助函数的导入。                                                                                                        |

### 行为变化

| 变更                                              | 具体变化                                                                                   |
| ----------------------------------------------- | -------------------------------------------------------------------------------------- |
| `max_age` 现在会被强制执行                              | 现在，请求新鲜登录的客户端会真正获得新登录，而不再被忽略。                                                          |
| Token introspection 保持一致                        | 不透明 token 返回与 JWT 相同的 claims，而且资源服务器可以 introspect 另一个客户端的 token。                       |
| ID token claims 保持协议安全                          | 自定义 claims 不再能覆盖保留的协议 claims，并且 ID token 会报告 `acr: "0"`。                               |
| 已授予的 scopes 会被保留                                | 后续登录不再会清除之前授予的 scopes。                                                                 |
| Google One Tap 更严格地验证 token                     | 格式错误的 identity token 会被拒绝并返回 `400`，`requireEmailVerification` 会返回 `403`。               |
| Magic-link 和 email-OTP 登录可以清除未经证实的凭据            | 在同一账户上，已验证的邮箱控制权会优先于未确认的密码。                                                            |
| `userinfo` 会用 401 拒绝错误的 access token            | 无效 token 会返回 `401 invalid_token`，并带有 `WWW-Authenticate` 头。                             |
| OAuth authorize 会将缺少 `response_type` 的错误重定向     | 错误会发送到已验证的客户端 `redirect_uri`，而不是通用错误页。                                                 |
| OAuth 客户端创建返回 `201`                             | 创建客户端会返回 `201 Created` 而不是 `200 OK`，并且注册过程会强制执行相同的权限检查。                                |
| Drizzle adapter 会验证受影响行数                        | 无效的受影响行数会直接抛出错误，而不是返回 `0`。                                                             |
| `organization.updateTeam` 会忽略不可变字段              | 请求体中不再接受 `id`、`createdAt` 和 `updatedAt`。                                               |
| `updateMemberRole` 会先检查授权                       | 角色是否存在的校验会在授权检查之后执行。                                                                   |
| CLI `generate --output` 到目录                     | 会选择与适配器相关的默认文件名。                                                                       |
| 生成的 schema 会跳过禁用迁移的模型                           | 会省略对已禁用迁移的模型的引用。                                                                       |
| Cookie-cache session 绑定到其 cookie                | 缓存的 session 会绑定到 `session_token` cookie。                                               |
| SSRF 主机检查覆盖更多保留范围                               | 出站主机分类现在会阻止额外的保留范围（6to4 relay anycast、site-local IPv6，以及 IPv4-compatible IPv6）。        |
| OAuth token 兑换错误使用标准代码                          | 授权码兑换失败会返回 `400 invalid_grant`，而不是 `401 invalid_client` 或 `invalid_request`。           |
| Sign-out 在外部 session 存储下运行 session-delete hooks | 即使使用 `secondaryStorage` 和 `preserveSessionInDatabase`，`session.delete` hooks 也会在登出时运行。 |
| 双因素失效失败有了自己的错误码                                 | 失败的双因素 challenge 清理现在会返回 `FAILED_TO_INVALIDATE_TWO_FACTOR_CHALLENGE`。                  |

### 新增内容

| 变更                      | 现在可用的内容                                                                |
| ----------------------- | ---------------------------------------------------------------------- |
| 原生客户端的刷新令牌重试            | 在短暂的重用窗口内重放相同的刷新响应。                                                    |
| OAuth 提供程序扩展面           | 添加授权类型、客户端身份验证方法、发现元数据和声明。                                             |

| 每次请求的登录选项               | 在一次登录请求中传递特定于提供程序的登录选项。                                                |
| 证书和签名断言登录               | 对 Microsoft Entra ID 使用证书登录，并对通用 OAuth 使用签名 JWT。                       |
| Private-key-JWT 客户端认证   | 使用签名 JWT（RFC 7523）而不是共享密钥来认证 OAuth 提供程序和 SSO 客户端。                      |
| SCIM 组                  | 使用持久的生命周期端点管理 SCIM 组。                                                  |
| 请求特定的用户声明               | 使用 `claims.userinfo` 参数请求单个用户详情。                                       |

| 会话和工具                   | 使用公钥会话验证、`hydrateSession`、i18n 和 `create-admin`。                       |

## 迁移到 v1.7

由于此版本包含较大的变更，我们会先以候选发布版的形式提供，因此会有一个迁移窗口。我们会继续关注反馈，并在此基础上发布稳定版 1.7。

请从 `rc` 标签安装它，并对你使用的任何 `@better-auth/*` 包也执行同样操作：

```package-install
better-auth@rc
```

然后对任何架构变更运行 `npx auth@rc generate`；未加作用域的 `npx auth` 目前仍会解析到 1.6 版 CLI，直到 1.7 稳定发布。下表显示这些变更是否会影响你。有关详细迁移指南，请参阅 [1.7 升级指南](/docs/guides/1-7-upgrade-guide)。

| 如果你使用                         | 预期情况                                          |
| ----------------------------- | --------------------------------------------- |
| 基础 Better Auth 配置             | 通常只需要安装 `@rc` 版本以及任何生成的架构变更                   |
| 社交登录、通用 OAuth、One Tap 或 SSO   | 新的登录客户端行为，尤其是通用 OAuth、身份令牌验证和 Google One Tap  |
| `@better-auth/oauth-provider` | 变更最多：资源、DPoP、反向通道注销、`max_age`，以及更安全的 OAuth 检查 |
| MCP                           | 包迁移到 `@better-auth/mcp`、新的导入、新的端点路径，以及架构迁移    |
| SAML 或 SCIM                   | 更安全的 SAML 和 SSO 检查，以及手动的 SCIM 所有权清理           |
| Magic link 或邮箱 OTP            | 对邮箱从未确认过的账户进行更安全的处理                           |
| 自定义适配器、存储或限流存储                | 需要新的原子方法                                      |
| 自定义代理或 TLS 终止                 | 在依赖 DPoP 或身份提供方重定向之前，检查你的应用如何计算其公共源地址         |
